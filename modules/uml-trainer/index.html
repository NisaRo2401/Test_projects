Anforderungsanalyse

Anwendungsfalldiagramm

Zeigt, welche Akteure mit dem System interagieren und welche Funktionen aus fachlicher Sicht angeboten werden.

Akteur Kunde --> (Ticket buchen) Akteur Admin --> (Benutzer verwalten)
  • Akteure stehen ausserhalb des Systems.
  • Use Cases beschreiben Ziele, keine technischen Klickfolgen.
  • Systemgrenze hilft, Verantwortlichkeiten sichtbar zu machen.
Gut fuer: Anforderungen, Rollen, fachliche Funktionen
Typischer Fehler: Datenbanktabellen oder Buttons einzeichnen
Prozesssicht

Aktivitaetsdiagramm

Zeigt einen Ablauf mit Entscheidungen, Schleifen und parallelen Schritten. Ideal, wenn du einen Prozess erklaeren sollst.

Start -> Login pruefen -> [gueltig?] Ja -> Dashboard anzeigen Nein -> Fehlermeldung anzeigen Ende
  • Start- und Endknoten geben den Rahmen.
  • Entscheidungen brauchen klare Bedingungen.
  • Parallele Verarbeitung wird mit Synchronisationsbalken modelliert.
Gut fuer: Workflows, Fachprozesse, Alternativen
Typischer Fehler: Klassen oder Datenstrukturen statt Aktivitaeten zeigen
Struktursicht

Klassendiagramm

Beschreibt statische Struktur: Klassen, Attribute, Methoden und Beziehungen. In der Pruefung oft fuer Domaenen- oder Entwurfsmodelle wichtig.

Kunde -------------------- - kundennummer: int - name: String -------------------- + bestellungAnlegen()
  • Attribute beschreiben Zustand, Methoden Verhalten.
  • Assoziationen zeigen Beziehungen zwischen Klassen.
  • Multiplizitaeten wie 1, 0..1 oder 1..* sind oft Punktebringer.
Gut fuer: Objektstruktur, Verantwortungen, Beziehungen
Typischer Fehler: Ablauf in ein Klassendiagramm pressen
Interaktion

Sequenzdiagramm

Zeigt zeitliche Reihenfolge von Nachrichten zwischen Objekten oder Komponenten. Perfekt fuer "wer ruft wen wann auf?".

Benutzer -> UI: login() UI -> AuthService: pruefeLogin() AuthService -> Datenbank: ladeBenutzer() Datenbank --> AuthService: Benutzerdaten
  • Zeit verlaeuft von oben nach unten.
  • Lebenslinien stehen fuer Beteiligte.
  • Nachrichten benoetigen sinnvolle Operationsnamen.
Gut fuer: Aufrufreihenfolgen, Schichten, Systemzusammenspiel
Typischer Fehler: Nur Klassenbeziehungen ohne Nachrichten darstellen

Musterloesung

So funktioniert der Builder

Akteur, Aktivitaet, Klasse oder Teilnehmer aus der Palette ziehen.
Element anklicken und oben den Namen setzen, zum Beispiel "Kunde" oder "Bestellung".
Fuer Kanten zuerst "Verbinden" aktivieren, dann Quelle und Ziel anklicken.
Die Pruefung achtet auf passende Symboltypen, wichtige Beschriftungen und erwartete Beziehungen.

Die Seite speichert deinen Fortschritt lokal im Browser. Sie ist bewusst auf pruefungsnahe Kernmuster reduziert, damit du UML schneller erkennst und sicherer einsetzt.
